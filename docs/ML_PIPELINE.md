## Eco‑Nexus ML Pipeline

### 1. Purpose

The ML pipeline provides **sustainability‑aware scoring** of vendor proposals and feeds the embedded decision engine with:

- A **numerical score** (0–100),
- Optional **confidence metrics**, and
- Explanatory metadata for debugging and analysis.

The goal is to reflect realistic ML workflows while keeping the stack lightweight and reproducible.

---

### 2. Data Model

Each proposal is represented by a feature vector such as:

- `price_today` – current vendor price for the contract.
- `delivery_days` – projected delivery / completion time.
- `local_flag` – whether the vendor is local or remote.
- `past_sustainability_avg` – historical sustainability performance.
- Optional:
  - `reliability_score`
  - `carbon_intensity_index`
  - `on_time_ratio`

Raw data is ingested from:

- CSV/JSON files (e.g., generated by `scripts/generate_mock_data.py`),
- or external systems (if integrated later).

---

### 3. Training Workflow

Located in `ml/train_model.py`.

High‑level steps:

1. **Load Data**
   - Read CSV/JSON input.
   - Validate schema and handle missing values.

2. **Feature Engineering**
   - Normalize or standardize numeric features.
   - Encode categorical flags.
   - Optionally derive new features (e.g., `price_per_day`).

3. **Model Selection**
   - Start with tree‑based models (e.g., `RandomForestRegressor`).
   - Optionally compare with:
     - Gradient boosting,
     - Linear models with regularization.

4. **Evaluation**
   - Train/validation split.
   - Metrics:
     - MAE / MSE,
     - R² score,
     - Calibration checks (score distributions).

5. **Export**
   - Save the final model as a `.joblib` artifact.
   - Persist metadata (features used, training date, metrics).

This pipeline is intentionally simple but structured to reflect **real ML practice**.

---

### 4. Inference & Integration

Inference is handled by `ml/predict_sustainability.py`, which:

- Loads the trained model (`.joblib`).
- Accepts feature values via:
  - Command‑line arguments, or
  - A small wrapper function for in‑process calls.
- Returns:
  - A clamped score in `[0, 100]`,
  - Optionally a JSON blob with intermediate details.

Calls can originate from:

- The Python **orchestration layer** (most common).
- The **C engine** via a sidecar process or bindings.
- Legacy systems (e.g., Node.js) if needed.

---

### 5. Reproducibility

To keep experiments reproducible:

- Model training scripts accept:
  - Random seeds,
  - Configurable hyperparameters,
  - Explicit train/test splits.
- Outputs include:
  - Training config,
  - Git commit hash (if available),
  - Dataset checksum.

Running `scripts/run_all_linux.sh` will:

- Install required Python packages from `ml/requirements.txt`.
- Train (or reuse) a model.
- Log where artifacts are stored.

---

### 6. Example Usage

Train a model:

```bash
cd ml
python3 train_model.py --data ../data/proposals.csv --out model.joblib
```

Run a single prediction:

```bash
python3 predict_sustainability.py model.joblib 100.0 7 1 82.5
```

Where the arguments are:

1. Model path
2. `price_today`
3. `delivery_days`
4. `local_flag` (0 or 1)
5. `past_sustainability_avg`

This interface is kept intentionally simple so it can be called from many environments.

---

### 7. Mapping to Project Goals

The ML pipeline:

- Provides a **clear, inspectable implementation** of the “AI/ML algorithms” in the project description.
- Integrates tightly with the embedded engine via:
  - Well‑defined interfaces,
  - Reproducible artifacts.
- Supports meaningful experiments that connect:
  - **Model quality**, and
  - **End‑to‑end decision latency** and throughput.
